## Phase D: 並行テスト（5/15〜6/15）

### 運用ルール: 二重更新

この期間中、コンテンツ更新は**両方のサイトに反映**する。

| 更新内容 | 旧サイト（HTML） | 新サイト（Markdown） |
|----------|-----------------|---------------------|
| 例会報告 | HTML作成 → FTPアップ | .md作成 → git push |
| お知らせ | index.htm編集 → FTPアップ | news/*.md作成 → git push |
| 行事予定 | HTML編集 → FTPアップ | events.md編集 → git push |

**新サイト側の更新を先に行い、旧サイトにコピーする流れが効率的。**
Markdownの方が編集しやすいため、新→旧の順番にする。

### テスト項目
- [ ] 全ページの表示確認（PC/スマホ）
- [ ] 画像の表示確認
- [ ] 内部リンクの動作確認
- [ ] OGP・SNSシェア確認
- [ ] Google検索からの到達確認
- [ ] Lighthouse: Performance/SEO/Accessibility 各90点以上

## 切替（6月中旬目標）

### 切替手順

1. **独自ドメイン取得**（まだの場合）
2. **新サイトにドメイン設定** → Cloudflare PagesでSSL自動
3. **旧サイト（さくら）にリダイレクト設置**
   ```html
   <!-- index.htm を以下に差し替え -->
   <meta http-equiv="refresh" content="0;url=https://新ドメイン/">
   ```
4. **会員への周知**（メール/会報/次回観察会で告知）
5. **さくらインターネット契約**は3ヶ月維持（リダイレクト用）→ その後解約

### 切替タイミングの根拠
- 6月の観察会（2026年度最初の野外行事）前に新サイトを公開
- 観察会で会員に直接新URLを案内できる
- 5月の総会でも告知可能

## 新サイトでの更新フロー（切替後）

### 例会報告の追加

```bash
# 1. 写真を配置
cp 観察会写真/*.jpg src/content/reikai/2026/images/

# 2. Markdownで報告を書く
vim src/content/reikai/2026/06-kawagoe.md

# 3. プッシュ → 自動デプロイ
git add -A && git commit -m "docs: 2026年6月川越観察会報告" && git push
```

### お知らせの更新

```bash
# 1. ニュース記事を作成
vim src/content/news/2026-06-report.md

# 2. プッシュ
git add -A && git commit -m "news: 観察会報告更新" && git push
```

**所要時間: 旧サイト（HTML編集+FTPアップ）と同等かそれ以下。**
**メリット: バージョン管理、自動バックアップ、どこからでも更新可能。**

## リスクと対策

| リスク | 対策 |
|--------|------|
| 二重更新の手間 | Phase Dを最短（1ヶ月）に抑える。新サイト→旧サイトの順で効率化 |
| 切替時のリンク切れ | 旧サイトにリダイレクト設置。3ヶ月維持 |
| 会員への周知不足 | 総会(5月)+観察会(6月)+メールで3回告知 |
| 2026年度行事が切替前に発生 | Phase A で既存サイトを先に更新済みなので問題なし |
| 写真の移行漏れ | ダウンロード済みバックアップ(632MB)から全て取得可能 |
